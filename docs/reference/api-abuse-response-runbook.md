# API Abuse Response Runbook

Last reviewed: 2026-04-03

이 문서는 KESE 재점검에서 남았던 운영 공백을 줄이기 위한 최소 대응 기준서다.
대상은 외부 API 쿼터를 소모하거나 비용이 큰 연산을 수행하는 엔드포인트다.

## 보호 대상 라우트

- `/api/extract-pdf`
- `/api/geo/address`
- `/api/geo/reverse`
- `/api/r2/upload/sign-pdf`
- `/api/r2/upload`
- `/api/profile/gallery`
- `/api/property/gallery`

## 자산 접근 원칙

- `pdf-temp/**` 는 임시 업로드 전용 경로다.
- `pdf-temp/**` 는 서명된 `PUT` 업로드와 서버 측 `GetObject` 처리만 사용한다.
- `pdf-temp/**` 는 클라이언트 응답에 퍼블릭 URL을 포함하지 않는다.
- 공개 URL이 필요한 경로는 현장 대표 이미지, 브리핑 커버, 프로필/갤러리 이미지처럼 서비스 노출이 의도된 비민감 이미지로 한정한다.
- 신분증, 계약서, 원본 PDF 같은 민감 파일은 퍼블릭 도메인 직접 노출을 금지한다.

## 기본 방어선

- 인증 필요 여부를 라우트별로 명시한다.
- 입력값은 `zod` 또는 동등 수준으로 검증한다.
- 민감 라우트는 `checkAuthRateLimit()` 또는 동등한 fail-secure 제한을 적용한다.
- 외부 API 프록시는 캐시 가능한 응답에 캐시를 둔다.
- 서버 로그에는 토큰, 키, 세션, 전화번호 원문을 남기지 않는다.

## 분류 기준

- 공개 허용: 정적 콘텐츠, 공개 게시물 조회처럼 비용이 낮고 민감하지 않은 조회
- 로그인 필요: 외부 API 쿼터를 소모하거나 사용자 데이터에 연결되는 조회/업로드
- 역할 필요: 관리자, 상담사, 회사 계정처럼 권한 검증이 필요한 변경 작업

## 이상 징후 판단

- 10분 이내 동일 사용자 또는 동일 IP에서 429 응답 급증
- Kakao, Naver, OpenAI, R2 같은 외부 제공사 쿼터 급감
- PDF 분석 요청이 평시 대비 급증
- 임시 파일 정리 실패로 `pdf-temp/` 오브젝트가 비정상적으로 누적
- 인증 실패, 권한 실패, 업로드 실패가 단시간에 반복

## 대응 절차

1. 영향 범위를 확인한다.
2. 최근 15분 로그에서 라우트, 사용자 ID, IP, 요청량, 실패율을 확인한다.
3. 문제 라우트의 rate limit 값을 즉시 강화하거나 임시 차단한다.
4. 외부 키 유출 의심 시 해당 공급자 키를 회전한다.
5. `pdf-temp/` 관련 사고면 만료 전 임시 파일 정리 라우트와 R2 오브젝트 상태를 확인한다.
6. 사용자 데이터 노출 가능성이 있으면 운영 담당자와 개발 담당자에게 즉시 공유한다.
7. 사고 종료 후 재발 방지 조치를 문서화한다.

## 역할 분담

- 개발 담당자: 라우트 차단, rate limit 조정, 키 회전, 코드 수정
- 운영 담당자: 공급자 콘솔 확인, 쿼터 상태 확인, 알림 채널 전파
- 데이터 담당자: 저장 데이터 영향도 확인, 백업/복구 필요 여부 판단

## 점검 체크리스트

- [ ] 민감 라우트가 인증 또는 역할 검증 뒤에 있는가
- [ ] rate limit 키가 사용자 또는 IP 기준으로 안정적으로 잡히는가
- [ ] 공급자 콘솔에서 쿼터/과금 이상 여부를 확인했는가
- [ ] 민감 파일 경로에 퍼블릭 URL 반환이 없는가
- [ ] 사고 이후 임시 완화책과 영구 조치를 둘 다 기록했는가
