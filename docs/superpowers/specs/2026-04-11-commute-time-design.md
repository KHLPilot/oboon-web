# 출퇴근 시간 기능 설계 — 현장 비교 교통·입지 탭

**날짜**: 2026-04-11  
**범위**: 현장 상세 > 현장 비교 섹션 > 교통·입지 탭  
**작성 배경**: CBD 거리(지하철까지 직선거리 km) 수치가 사용자에게 의미가 없어, 실제 근무지까지의 출퇴근 시간으로 교체

---

## 결정 사항

| 항목 | 결정 |
|------|------|
| 근무지 단위 | 역(station) 또는 구(district) — 정확한 주소 불필요 |
| 이동 수단 | 대중교통 + 자가용 나란히 표시 |
| 노출 위치 | 교통·입지 탭에만 (상세 페이지 다른 영역 미포함) |
| 근무지 입력 위치 | 교통·입지 탭 상단, 2컬럼 공통 입력 |
| 근무지 저장 | 프로필 기본값 + 탭에서 임시 변경 가능 |
| 계산 방식 | 즉시: 직선거리 추정 / 클릭 시: 서버 API 실제 경로 |

  ↓
[근무지: 강남역 ▾]  ← 프로필 기본값 or 미설정 시 "근무지 선택"
  ↓

## 근무지 선택 UI

- 검색창 + 주요 업무지구 빠른 선택 조합
  - 주요 업무지구 예: 강남, 여의도, 광화문, 판교, 을지로, 잠실 등
  - 역 검색: 기존 Kakao 주소 API 연동 활용
- 선택 저장
  - 로그인 사용자: 프로필(`profiles` 테이블)에 `workplace_code`, `workplace_type`, `workplace_label` 저장
  - 비로그인: `localStorage`에만 저장
- 탭에서 변경한 값은 임시 상태 (저장 버튼 없이 자동 반영, 프로필 업데이트는 별도 확인 없이 디바운스 저장)

## 직선거리 추정 로직 (즉시 표시)

- 현장 좌표(`property_locations.lat`, `lng`) ↔ 선택한 역/구 중심 좌표 → Haversine 공식으로 직선거리 계산
- 시간 추정 계수 (서울 평균 속도 기반):
  - 대중교통: `거리(km) / 0.4 * 60` 분
  - 자가용: `거리(km) / 0.6 * 60` 분
- 표시: "약 XX분 (추정)" — 추정임을 명확히 표기
- 계산은 클라이언트에서 수행 (API 호출 없음)

## 서버 API — `/api/commute-time`

**트리거**: 사용자가 [정확히 알아보기] 클릭 시에만 호출

### 요청

```
GET /api/commute-time?propertyId=123&workplaceType=station&workplaceCode=강남역&workplaceLat=37.498&workplaceLng=127.027
```

### 처리

1. `propertyId`로 `property_locations`에서 현장 좌표 조회
2. Naver Directions 5 API 호출 (대중교통, 자가용 각각)
3. 인메모리 캐시: `propertyId + workplaceCode` 키, TTL 24h

### 응답

```json
{ "transit": 48, "car": 27, "unit": "분" }
```

### 보안

- 인증된 사용자만 호출 가능 (`requireAuthenticatedUser`)
- API 키(Naver Directions)는 서버에만 보관
- 레이트 리밋 적용

## 타입 변경

### `OfferingCompareItem` (features/offerings/domain/offering.types.ts)

- `distanceToCbd: string` 필드 제거
- `commuteEstimate?: { transitMin: number; carMin: number }` 추가 (직선거리 기반 추정값, 서버에서 계산)

### 프로필 (`profiles` 테이블)

마이그레이션으로 컬럼 추가:
- `workplace_type: text` — `'station' | 'district'`
- `workplace_code: text` — 역명 또는 구명 (예: "강남역", "강남구")
- `workplace_label: text` — 표시용 라벨
- `workplace_lat: float8`
- `workplace_lng: float8`
