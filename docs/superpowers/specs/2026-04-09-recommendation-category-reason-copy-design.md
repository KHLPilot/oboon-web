# 추천 결과 카테고리 설명문 개선 설계

작성일: 2026-04-09

## 목적

현재 추천 결과의 카테고리별 상세 문구는 계산 로직이 반환하는 원문을 거의 그대로 보여주고 있다. 이 방식은 내부 판정에는 유용하지만, 사용자 입장에서는 "왜 이런 평가가 나왔는지"를 이해하기 어렵다.

이번 설계의 목적은 추천 결과의 카테고리 설명을 사용자 친화적인 안내형 문구로 재구성해, 계산 근거는 유지하면서도 해석 가능성을 높이는 것이다.

## 현재 문제

- 자금 항목이 `계약금의 240% - 충분한 현금 보유`처럼 내부 계산 표현에 가깝다.
- 주택 보유, 구매 목적 항목이 `무주택`, `실거주` 같은 입력값 수준으로만 노출된다.
- 사용자는 "무엇을 입력했는지"는 알 수 있어도, "그래서 왜 이 점수가 나왔는지"는 알기 어렵다.
- 점수 계산 로직은 이미 충분히 존재하지만, UI에 전달되는 문구 레이어가 설명 역할을 하지 못한다.
- 가격 비공개 매물에 대해서는 자금 관련 수치를 그대로 드러내면 안 된다.

## 목표

- 카테고리별 상세 문구를 설명형으로 바꾼다.
- 톤은 부드럽지만 분석적인 안내형으로 맞춘다.
- 점수 숫자는 직접 노출하지 않는다.
- 낮은 평가 항목은 이유를 분명하게 설명한다.
- 수치화 가능한 항목은 설명 안에 수치를 포함한다.
- 단, 가격 비공개 매물은 자금 항목만 수치 비공개로 처리한다.

## 비목표

- 점수 계산 로직 자체를 바꾸지 않는다.
- 카테고리 구성이나 배점 체계를 바꾸지 않는다.
- 공개/비공개 정책을 새로 정의하지 않는다.
- 자금 외 항목까지 비공개 정책으로 확장하지 않는다.

## 현재 구조 요약

현재 추천 결과는 두 흐름으로 카테고리 결과를 만든다.

- 상세 조건 검증 카드
  - `features/offerings/components/detail/ConditionValidationCard.tsx`
  - API가 내려주는 `categories[key].reason`을 그대로 노출한다.
- 추천 리스트의 타입 결과
  - `features/recommendations/hooks/useRecommendations.ts`
  - `mapRecommendationUnitCategory()`에서 `raw.reason`을 그대로 `reason` 필드로 보존한다.

계산 원문은 아래 도메인 로직에서 만들어진다.

- 로그인 평가: `features/condition-validation/domain/fullCustomerEvaluator.ts`
- 비로그인 평가: `features/condition-validation/domain/guestEvaluator.ts`

현재 원문은 내부 기준을 반영하지만 사용자 설명으로는 불친절하다.

## 최종 방향

카테고리별 상세 문구는 더 이상 계산 원문을 직접 보여주지 않는다. 대신 "사용자용 설명문 생성 레이어"를 별도로 두고, UI는 이 레이어가 만든 설명문만 노출한다.

- 내부 계산 결과는 유지
- 내부 원문 `reasonMessage`는 설명문 생성의 입력으로만 사용
- 사용자에게는 해석된 문장만 노출
- 점수 숫자는 숨김
- 필요할 때만 보조 수치를 같은 문장 안에 포함

즉, 구조는 아래처럼 분리된다.

1. 계산 로직
점수와 원시 reason, metrics 생성

2. 설명문 생성 로직
카테고리별 사용자용 reason 생성

3. UI 렌더링
설명문만 노출

## 문구 원칙

### 1. 설명형

문구는 입력값 라벨이 아니라 판단 결과를 설명해야 한다.

좋은 예:

- `무주택 조건이라 현재 추천 기준에서 유리하게 반영됐어요.`
- `실거주 목적이라 현재 추천 기준과 잘 맞는 편으로 반영됐어요.`

나쁜 예:

- `무주택`
- `실거주`
- `계약금의 240% - 충분한 현금 보유`

### 2. 부드럽지만 분석적인 톤

문구는 친절해야 하지만, 막연하면 안 된다.

- 너무 가벼운 톤은 피한다.
- 너무 딱딱한 심사 문구도 피한다.
- "왜 이렇게 반영됐는지"가 드러나는 문장을 우선한다.

기준 예시:

- `현재 조건에서 유리하게 반영됐어요`
- `보수적으로 반영됐어요`
- `추천 기준과 잘 맞는 편이에요`
- `상대적으로 낮게 반영됐어요`

### 3. 낮은 평가는 분명하게

낮은 점수 구간은 완곡하게 돌리지 않는다.

예:

- `권장 자금 기준보다 약 2,400만원 부족해서 자금 항목 평가가 낮아졌어요.`
- `기존 주택 보유 수가 있어 해당 항목은 보수적으로 반영됐어요.`
- `투자 목적 비중이 커서 해당 항목 평가는 상대적으로 낮아졌어요.`

### 4. 점수 숫자는 숨김

- `8점`, `24/30` 같은 점수 수치는 보여주지 않는다.
- 사용자는 점수보다 해석을 이해해야 한다.

### 5. 수치화 가능한 항목은 설명에 포함

가능하면 설명문 안에 핵심 수치를 포함한다.

예:

- `권장 자금 기준보다 약 2,400만원 부족해서 자금 항목 평가가 낮아졌어요.`
- `희망 시점과 실제 일정 차이가 약 4개월 있어 시점 적합도가 낮아졌어요.`
- `예상 월 부담이 월소득의 42% 수준이라 상환 부담이 다소 높은 편으로 반영됐어요.`

수치는 별도 표보다 설명문 안에 자연스럽게 녹여 넣는다.

## 공개/비공개 정책

### 공개 매물

자금 항목은 설명 + 수치를 허용한다.

예:

- `권장 자금 기준을 충분히 넘어서 자금 여유가 안정적으로 반영됐어요.`
- `계약금은 가능하지만 권장 자금 기준보다 약 1,100만원 부족해 여유가 크지 않은 편으로 반영됐어요.`

### 비공개 매물

자금 항목만 수치를 숨긴다.

- 부족 금액
- 권장 금액
- 계약금 대비 비율

위 수치는 직접 노출하지 않는다.

예:

- `권장 자금 기준에 다소 못 미쳐 자금 항목이 보수적으로 반영됐어요.`
- `자금 여유가 충분한 편으로 반영됐어요.`

중요한 점은 "비공개 정책 때문에 숨겼다"는 설명을 따로 붙이지 않는 것이다. 사용자에게는 그냥 자연스러운 문장만 보인다.

### 다른 항목

자금 외 카테고리는 기존처럼 수치화 가능한 범위에서 수치를 계속 사용할 수 있다.

즉, 비공개 매물의 예외는 자금 항목에만 적용한다.

## 카테고리별 설계

### 자금력

#### 사용 데이터

- `availableCash`
- `contractAmount`
- `minCash`
- `recommendedCash`
- `isPricePublic`

#### 설명 방향

- 충분: 권장 기준을 넘는지 설명
- 경계: 계약금은 가능하지만 여유가 작은지 설명
- 부족: 권장 기준 대비 부족 금액을 설명
- 비공개 매물: 같은 판단을 수치 없이 일반화

#### 공개 매물 예시

- `권장 자금 기준을 충분히 넘어서 자금 여유가 안정적으로 반영됐어요.`
- `계약금은 가능하지만 권장 자금 기준보다 약 1,100만원 부족해 여유가 크지 않은 편으로 반영됐어요.`
- `권장 자금 기준보다 약 3,200만원 부족해서 자금 항목 평가가 낮아졌어요.`

#### 비공개 매물 예시

- `자금 여유가 충분한 편으로 반영됐어요.`
- `권장 자금 기준에 다소 못 미쳐 자금 항목이 보수적으로 반영됐어요.`
- `자금 여유가 부족해서 자금 항목 평가가 낮아졌어요.`

### 소득 / 상환여력

로그인/비로그인 흐름이 다르므로 계산식은 달라도 설명 구조는 통일한다.

#### 사용 데이터

- `monthlyPaymentEst`
- `monthlyBurdenPercent`
- 필요 시 `monthlyIncome`, `monthlySurplus`, DSR 정보

#### 설명 방향

- 부담이 낮으면 안정적으로 반영됐다고 설명
- 부담이 높으면 왜 낮아졌는지 분명히 설명
- 가능하면 `%` 수치 사용

#### 예시

- `예상 월 부담이 소득 대비 무리 없는 수준이라 안정적으로 반영됐어요.`
- `예상 월 부담이 월소득의 42% 수준이라 상환 부담이 다소 높은 편으로 반영됐어요.`

### LTV·DSR / 신용

로그인 상태에서는 `LTV·DSR`, 비로그인 상태에서는 `신용` 카테고리가 나온다. 설명 생성 로직은 각각 별도로 두되, 톤은 통일한다.

#### LTV·DSR 예시

- `대출 가능 범위와 부담 수준이 전반적으로 무난하게 반영됐어요.`
- `DSR 부담이 높아 대출 여건이 보수적으로 반영됐어요.`

#### 신용 예시

- `현재 신용 상태가 양호해 대출 관련 평가에 유리하게 반영됐어요.`
- `신용 조건이 다소 불리해 대출 관련 평가는 보수적으로 반영됐어요.`

### 주택 보유

#### 사용 데이터

- `houseOwnership`

#### 설명 방향

- 단순 라벨을 보여주지 않고 해석 문장으로 바꾼다.

#### 예시

- `무주택 조건이라 현재 추천 기준에서 유리하게 반영됐어요.`
- `1주택 조건이라 일부 기준에서 보수적으로 반영됐어요.`
- `기존 주택 보유 수가 있어 해당 항목 평가는 낮아졌어요.`

### 구매 목적

#### 사용 데이터

- `purchasePurpose`

#### 설명 방향

- 목적 값이 아니라 추천 기준과의 적합도를 설명한다.

#### 예시

- `실거주 목적이라 현재 추천 기준과 잘 맞는 편으로 반영됐어요.`
- `장기 보유 목적이라 실거주와 투자 성향이 함께 반영됐어요.`
- `투자 목적 비중이 커서 해당 항목 평가는 상대적으로 낮아졌어요.`

### 시점

#### 사용 데이터

- 청약/계약/입주 관련 일정
- 사용자 희망 시점과의 차이

#### 설명 방향

- 차이가 작으면 잘 맞는다고 설명
- 차이가 크면 개월 수를 포함해 설명

#### 예시

- `희망 시점과 실제 일정 차이가 크지 않아 무난하게 반영됐어요.`
- `희망 시점과 실제 일정 차이가 약 4개월 있어 시점 적합도가 낮아졌어요.`

## 설명문 생성 구조

설명문 생성은 계산 로직 내부에 섞지 않고 별도 유틸 레이어로 둔다.

권장 구조:

- `features/recommendations/lib/`
- 또는 `features/condition-validation/lib/`

역할:

- raw category result
- raw metrics
- 사용자 입력 condition
- 공개/비공개 여부

위 값을 받아 사용자용 설명문을 만든다.

이 레이어는 UI에 직접 종속되지 않아야 하며, 추천 리스트와 상세 조건 검증 카드에서 공통으로 재사용할 수 있어야 한다.

## 데이터 모델 변경 방향

현재 `reason`은 계산 원문을 그대로 들고 있다. 이를 아래 둘 중 하나로 정리한다.

### 옵션 A

- `reason`: 사용자용 설명문으로 교체
- 내부 원문은 버림

### 옵션 B 권장

- `rawReason`: 내부 계산 원문
- `reason`: 사용자용 설명문

권장 이유:

- 디버깅과 비교가 쉽다.
- UI는 `reason`만 사용하면 된다.
- 필요 시 내부 분석용으로 `rawReason`을 보존할 수 있다.

## UI 반영 원칙

### 상세 조건 검증 카드

`ConditionValidationCard`는 `cat.reason`을 직접 출력하는 구조이므로, 이 필드가 사용자용 설명문을 받도록 연결한다.

### 추천 리스트의 타입 결과

`mapRecommendationUnitCategory()`에서 사용자용 설명문을 채워 넣는다. 타입 패널과 시트는 별도 수정 없이 `reason`을 그대로 쓰게 만들 수 있다.

단, 현재 타입 패널은 카테고리 라벨 중심으로 보이고 reason은 충분히 노출되지 않는다. 구현 단계에서는 아래 둘 중 하나를 선택해야 한다.

- 카테고리 확장 상태에서 설명문 노출
- 상위 1~2개 약점 카테고리 설명문만 요약 노출

이번 설계의 핵심은 문구 시스템이며, 패널 UI 확장은 문구가 실제 읽히도록 하는 최소 범위에서 함께 다룬다.

## 테스트 관점

테스트는 문구가 "어떤 기준으로 번역되는지"를 보호해야 한다.

필수 테스트:

- 자금력 공개 매물: 부족 금액이 포함된 설명 생성
- 자금력 비공개 매물: 부족 금액 미노출
- 주택 보유: `무주택` 같은 라벨이 직접 reason으로 남지 않음
- 구매 목적: `실거주` 단어만 던지는 대신 적합도 설명 생성
- 시점: 수치화 가능한 경우 개월 수 포함

즉, 단순 문자열 존재 검사보다 "문구 생성 규칙"을 직접 검증해야 한다.

## 구현 범위

이번 변경에 포함할 범위:

- 사용자용 설명문 생성 유틸 추가
- 추천 리스트 타입 결과 reason 재매핑
- 상세 조건 검증 카드 reason 재매핑
- 관련 테스트 추가

이번 변경에 포함하지 않을 범위:

- 배점 체계 수정
- API 응답 스키마 대수술

## 결정 사항 요약

- 문구는 설명형으로 간다.
- 톤은 부드럽지만 분석적인 안내형으로 간다.
- 점수 숫자는 직접 보여주지 않는다.
- 낮은 평가는 이유를 분명하게 설명한다.
- 수치화 가능한 항목은 설명 안에 포함한다.
- 비공개 매물은 자금 항목만 수치 비공개로 처리한다.
- 비공개 여부는 문구에서 따로 밝히지 않는다.
- 계산 로직은 유지하고, 설명문 생성 레이어를 별도로 둔다.
