# 분양가표 레이아웃 재설계

작성일: 2026-04-09

## 목적

현장 상세의 분양가표는 조건 검증을 하지 않은 사용자도 타입별 기본 정보를 빠르게 비교할 수 있어야 한다. 동시에 조건 검증을 완료한 사용자에게는 같은 표 안에서 개인화 결과를 자연스럽게 덧입혀 보여줘야 한다.

이번 설계의 목적은 분양가표를 "조건 결과 중심 UI"가 아니라 "가격 중심 기본 정보표 + 조건 결과 보조 레이어"로 재정렬하는 것이다.

## 문제 정의

- 현재 타입별 UI는 조건 결과와 평면도 비중이 커서 분양가표의 기본 역할이 약하다.
- 조건 검증을 하지 않은 사용자는 타입별 가격과 스펙 비교 흐름을 충분히 얻기 어렵다.
- 조건 검증 결과를 강하게 전면 배치하면 가격 비교 흐름이 끊긴다.
- 모바일에서 테이블형 정보와 상세형 정보를 동시에 강하게 노출하면 밀도가 급격히 높아진다.

## 목표

- 조건 검증 전에도 타입별 가격 비교가 가능해야 한다.
- 가격은 항상 가장 먼저 읽히는 핵심 정보여야 한다.
- 전용면적은 가격 다음 우선순위의 보조 정보로 유지한다.
- 조건 검증 결과는 기존 표를 대체하지 않고 보조 레이어로 덧붙인다.
- 데스크톱과 모바일 모두 같은 정보 구조를 유지하되 표현 밀도만 조정한다.

## 비목표

- 분양가표를 추천 카드나 상담 유도 전용 UI로 바꾸지 않는다.
- 타입별 모든 상세 정보를 접힌 상태에서 항상 노출하지 않는다.
- 조건 검증을 하지 않은 상태에서 결과성 문구나 비어 있는 결과 칸을 강제로 보여주지 않는다.

## 최종 방향

분양가표는 `행 요약 + 확장형` 구조로 재설계한다.

- 접힌 기본 행은 누구나 읽을 수 있는 공용 정보 레이어다.
- 펼침 상세는 가격 상세, 타입 정보, 조건 검증 상세를 함께 담는 확장 레이어다.
- 조건 검증을 하지 않은 사용자는 기본 행만으로도 비교가 가능해야 한다.
- 조건 검증을 한 사용자는 같은 행 안에서 개인화 보조 정보가 추가된다.

핵심 우선순위는 아래 순서를 따른다.

1. 정확한 분양가 범위
2. 타입명
3. 전용면적
4. 보조 스펙
5. 조건 결과 또는 조건 확인 CTA
6. 펼침 상세

## 기본 행 구조

각 타입 행은 접힌 상태에서 아래 구조를 가진다.

1. 타입명
2. 정확한 분양가 범위
3. 전용면적 중심 보조 정보
4. 평면도 보기 및 펼치기 액션
5. 조건 상태 행

예시 구조:

- `59A`
- `5.8억 ~ 6.1억`
- `전용 59㎡ · 3룸 · 2욕실 · 84세대`
- `평면도 보기`
- 미검증: `내 조건으로 확인`
- 검증 후: `우선 검토 · 초기 필요 자금 1.4억 · 월 부담률 32%`

이 구조에서 가격은 가장 크게 보이고, 전용면적은 보조 정보 첫 번째 토큰으로 고정한다.

## 조건 상태 레이어

조건 상태 레이어는 기본 행의 하단 보조층이다.

### 조건 검증 전

- 결과 배지를 노출하지 않는다.
- 빈 영역 대신 짧은 CTA를 보여준다.
- 권장 문구는 `내 조건으로 확인`으로 통일한다.

### 조건 검증 후

- 같은 위치에 결과 요약을 노출한다.
- 결과 요약은 아래 정보만 사용한다.
  - 결과 배지
  - 초기 필요 자금
  - 월 부담률
  - 한 줄 해석 요약

권장 결과 라벨:

- `우선 검토`
- `상담 후 검토`
- `조건상 부담 큼`

이 레이어는 가격보다 강하면 안 되며, 사용자가 "왜 이런 결과인지 더 보려면 펼치면 된다"는 역할만 수행한다.

## 펼침 상세 구조

행을 펼쳤을 때는 아래 순서로 정보를 노출한다.

### 1. 가격 정보

- 분양가 최소/최대
- 가격 공개 여부
- 필요한 경우 가격 관련 보조 문구

### 2. 내 조건 기준

- 계약금
- 최소 필요 현금
- 권장 현금
- 예상 월 납입액
- 월 부담률

이 섹션은 조건 검증 완료 사용자에게만 노출한다.

### 3. 판정 이유

- 자금력
- 대출 여건 또는 소득/부담
- 주택 보유
- 구매 목적
- 시점

각 항목은 작은 상태 칩과 짧은 설명 문장으로 보여준다. 숫자 중심 표보다 "어느 항목이 강점이고 약점인지"를 빠르게 읽게 하는 구성이 우선이다.

### 4. 타입 정보

- 전용면적
- 공급면적
- 방/욕실
- 구조
- 향
- 세대수

### 5. 평면도

- 평면도 이미지는 가장 마지막에 둔다.
- 평면도는 1차 비교 정보가 아니라 최종 검토 보조 정보로 취급한다.

## 데스크톱 배치

데스크톱에서는 한 행 안에서 좌우 위계를 분명히 둔다.

### 접힌 상태

- 좌측: 타입명
- 중앙: 정확한 분양가 범위
- 중앙 하단: `전용 59㎡ · 3룸 · 2욕실 · 84세대`
- 우측: 평면도 보기, 펼치기
- 하단 보조행: CTA 또는 조건 결과

가격은 시각적 중심이며, 조건 결과는 하단 보조행에 위치한다.

### 펼침 상태

2열 구조를 사용한다.

- 좌측 열: 가격 정보, 내 조건 기준, 판정 이유
- 우측 열: 타입 정보, 평면도

이 구조는 개인화 정보와 상품 정보를 분리해 읽기 쉽게 만든다.

## 모바일 배치

모바일에서는 테이블보다 압축된 리스트 카드처럼 보이게 구성한다.

### 접힌 상태

1줄째: 타입명, 조건 결과 배지 또는 CTA
2줄째: 정확한 분양가 범위
3줄째: `전용 59㎡ · 3룸 · 84세대`
4줄째: 검증 후일 때만 `초기 필요 자금 · 월 부담률`

모바일에서도 가격이 가장 크게 보여야 한다.

### 펼침 상태

세로 순서를 아래와 같이 고정한다.

1. 한 줄 요약
2. 가격 정보
3. 내 조건 기준
4. 판정 이유
5. 타입 정보
6. 평면도

평면도를 상단으로 올리지 않는다. 가격과 조건 정보가 스크롤 초반에 먼저 보여야 한다.

## 문구 원칙

### 기본 행

- 타입명은 짧고 그대로 사용한다. 예: `59A`
- 분양가 범위는 정확한 최소~최대 범위를 노출한다.
- 전용면적은 항상 `전용 59㎡` 형식으로 표기한다.

### 미검증 상태

- `내 조건으로 확인`

### 검증 상태 예시

- `우선 검토 · 초기 필요 자금 1.4억 · 월 부담률 32%`
- `자금 여건은 무난하지만 대출 조건은 확인이 필요합니다.`

문구는 확정 판정처럼 보이지 않도록 유지한다.

## 상태 전환 원칙

- 조건 검증 전후에 레이아웃 구조는 바뀌지 않는다.
- 같은 위치에 CTA가 결과 요약으로 치환된다.
- 조건 검증을 했다고 해서 가격 정보의 위치나 위계가 변하면 안 된다.
- 펼침 상세는 접힌 상태에서 이미 보이는 내용을 반복하지 말고, 이유와 수치 설명을 보강해야 한다.

## 접근성 및 상호작용 규칙

- 각 타입 행은 아코디언 패턴으로 동작한다.
- 펼침 버튼은 별도 버튼 역할과 상태를 가져야 한다.
- 평면도 보기 버튼은 펼침과 다른 액션으로 분리한다.
- 조건 검증 CTA는 버튼 역할을 가져야 한다.
- 접힘/펼침 상태는 스크린리더에 전달되어야 한다.

## 구현 범위 예상

주요 변경 대상:

- `features/offerings/components/detail/offeringTypesAccordion.client.tsx`
- `features/offerings/components/detail/ConditionValidationCard.tsx`
- 필요 시 타입별 조건 표시 유틸과 스타일 토큰

이번 변경은 데이터 계산 로직을 바꾸지 않고, 현재 제공되는 가격/타입/조건 결과 데이터를 레이아웃 우선순위에 맞게 재배치하는 것을 우선한다.

## 완료 기준

- [ ] 조건 검증 전에도 타입별 가격 비교가 가능하다.
- [ ] 가격이 각 타입 행에서 가장 먼저 읽히는 정보다.
- [ ] 전용면적이 보조 정보의 첫 번째 항목으로 노출된다.
- [ ] 조건 검증 전에는 CTA, 검증 후에는 결과 요약이 같은 위치에 노출된다.
- [ ] 펼침 상세에서 가격 정보, 조건 정보, 타입 정보, 평면도를 구조적으로 구분한다.
- [ ] 데스크톱과 모바일 모두 같은 정보 위계를 유지한다.
